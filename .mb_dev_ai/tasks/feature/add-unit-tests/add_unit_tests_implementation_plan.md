## 6. Design Decisions

| Decision | Options Considered | Chosen | Rationale |
| -------- | ------------------ | ------ | --------- |
| **Flutter utilities testing** | A: Widget tests (testWidgets) / B: Pseudo-unit tests with mocks | B: Pseudo-unit tests with mocks | Widget tests require Flutter environment and are slower. Pseudo-unit tests with mocktail are faster, sufficient for utilities, and align with requirement for single `dart test` command execution. |
| **Test organization** | A: One giant test file per package / B: Mirrored folder structure / C: Flat test directory | B: Mirrored folder structure | Mirrors lib/ structure making tests easy to locate and maintain. Clear organization helps identify which tests cover which source files. |
| **Time-dependent testing** | A: Use real DateTime.now() / B: Use fake_async package | B: fake_async package | Allows deterministic testing of time-dependent code. Using real DateTime.now() makes tests flaky and harder to reproduce. |
| **Coverage target** | A: 100% coverage / B: 80% coverage / C: 60% coverage | B: 80% coverage | 80% is achievable for utility code and provides good protection. 100% is often not cost-effective (testing private implementation details). |
| **Mock library** | A: mockito / B: mocktail / C: hand-rolled mocks | B: mocktail | Mocktail is null-safe and recommended for Dart 3+. Mockito requires code generation. Hand-rolled is more maintenance. |

### Decision Log: Flutter Utilities Testing Approach

**Context**: Flutter utilities (BuildContext, Widget extensions) require framework dependencies. Choosing the right testing approach impacts test speed, reliability, and CI/CD integration.

| Option | Pros | Cons | When to prefer |
|--------|------|------|----------------|
| A: Widget tests (testWidgets) | Tests with real Flutter framework; catches framework integration issues | Slower; requires Flutter test environment; cannot run with `dart test` | When testing UI widgets, animations, or complex widget interactions |
| B: Pseudo-unit tests with mocks | Fast; runs with `dart test`; deterministic; good for utility logic | Doesn't test real framework behavior; mocks may drift from actual APIs | When testing pure logic utilities that happen to use Flutter types |

**Decision**: B: Pseudo-unit tests with mocks — These are utilities, not UI widgets. The logic is simple (getters, simple calculations) and doesn't warrant the overhead of widget tests. Mocktail provides sufficient isolation.
